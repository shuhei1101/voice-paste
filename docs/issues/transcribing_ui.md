# 文字起こし中のUI表示

## やりたいこと

録音確定後、ウィンドウを閉じずに「文字起こし中...」＋経過時間を表示したい。

## 現状の問題

バックグラウンドスレッドで `transcriber.transcribe()` を実行すると、
2回目の呼び出しでCUDA/CTranslate2がクラッシュする（プロセスごと死亡）。

原因: CTranslate2のCUDAコンテキストがスレッド間で共有できない。
1回目のスレッドでCUDAコンテキストが初期化され、2回目の別スレッドから
同じモデルを使おうとしてsegfaultが発生する。

## 試したこと

- `root.quit()` + `show()` 末尾で destroy → 効果なし
- `shutil.copy2` → `shutil.move` → 効果なし（別の問題は解決）
- TclError ガード → 効果なし

## 根本原因の可能性

Tkのmainloopが生きたまま文字起こしを実行し、`root.quit()` → `root.destroy()`
→ 2回目の `tk.Tk()` 作成という流れが、Windows + CUDA環境で何かを壊している。
元のコード（文字起こし中画面なし）は `_confirm` 内で即 `destroy()` して
mainloopを抜けた後にメインスレッドで同期的に文字起こしするので問題が起きない。

つまりバックグラウンドスレッド単体の問題ではなく、
**Tk mainloop が生きている状態での CUDA 処理** が問題の可能性が高い。

## 解決アプローチ候補

1. **専用ワーカースレッドを1つ使い回す**: スレッドを毎回作るのではなく、
   起動時に1つだけワーカースレッドを作り、Queueで文字起こしリクエストを
   送る。同じスレッドでCUDAを使い続けるのでコンテキスト問題が起きない。

2. **メインスレッドで文字起こし + root.update() で UI更新**:
   Tkのmainloopを使わず、transcribe中にroot.update()を定期的に呼んで
   UIを更新する。ただしtranscribeがジェネレータなのでセグメント単位で
   update()を挟める可能性あり。

3. **別プロセスで文字起こし**: multiprocessingで別プロセスを立ち上げて
   transcribeを実行。CUDAコンテキストが完全に分離される。ただし
   モデルロードが毎回発生するので遅い。
