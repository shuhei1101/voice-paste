# PR11: 文字起こし後バックグラウンドでモデル再ロードしGPUメモリを管理

## 概要

PR10のセッションカウンター方式では、長い録音が続くと閾値（30回）に達する前に
GPUメモリが枯渇してクラッシュする問題が残っていた。

根本原因: CTranslate2のCUDAアロケーターがセッションをまたいでGPUメモリを
キャッシュし続けるため、累積処理音声時間が約2000秒を超えるとOOMクラッシュが発生。

## 対応方針

文字起こし完了直後に `threading.Lock` を使ってバックグラウンドスレッドで
モデルを再ロードする（del → gc.collect() → _load_model()）。

- 再ロードはアイドル時間（ユーザーが貼り付け結果を使っている間）に完了する
- 次の録音確定時に再ロードがまだ進行中なら lock 待ち（最大3秒）で対応
- セッションカウンター・WHISPER_REINIT_INTERVAL は廃止してシンプル化
